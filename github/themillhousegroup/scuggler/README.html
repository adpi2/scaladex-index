scuggler

Background

xuggler/humble-video are great Java libraries for video processing but they feel little "odd" when using from Scala; they use code structures and mechanisms that really don't feel very "idiomatic". Scuggler attempts to make xuggler/humble-video feel like just-another Scala library, and offer the following functionality:

  • Define a Frame consisting of a Long timestamp and a BufferedImage image
  • Be able to convert a File to a sequence of Frames
  • Be able to convert an InputStream to a sequence of Frames

Going "the other way" (i.e. being able to create a media file or stream from a sequence of Frames) is not in scope for 0.x releases.

Notes

This project is undergoing a transition from the Xuggler library which is now unsupported, to the Humble-Video package. When this has been completed satisfactorily, the version number will be incremented to 0.3.x. Scuggler's own API will not change.

This project is about to start using the Play Iteratee framework as it solves a number of substatial producer-consumer-efficiency problems.

Usage

The main thing you'll be interested in as a Scuggler client is the allFrames member of a FrameStreamer instance.

This is defined as:

  def allFrames: Future[Seq[Future[Frame]]]

which looks a little bit scary, but really is just a way of representing the fact that each frame takes a finite amount of time (and processing) to be created, and so if there are other things that can be done while we wait, they should be able to run. The "outer" Future will only complete when the final number of frames is known - allowing the Seq to be immutable.

Performance

There is an underlying belief that due to the use of scala's concurrent support, code being driven from scuggler rather than vanilla xuggler should make better use of available computing resources and should be at least as performant. As of Scuggler 0.1.x, here are some numbers (in Frames Per Second) when the libraries are used idiomatically:

Task Xuggler Scuggler (0.1.x)
No-Load 247 243
Average Pixel Colors 168 228
Task Explanations:

No-Load: When notified that a frame is ready, a counter is incremented. No work is done on the frame.

Average Pixel Colors: When notified that a frame is ready, the RGB triple of each pixel is examined, and the average pixel color for the frame is stored. This could be considered a "moderate" amount of per-frame processing.

For more detail on performance tests, or to run the tests on your own hardware, check PerformanceComparisonSpec.
